## Endpoint

```
GET /api/export/providers
```

Exports provider configurations including their associated tools and parameters in a portable JSON format suitable for backup, sharing, or migration.

## Query Parameters

<ParamField query="ids" type="array">
  Comma-separated list of provider IDs to export. If omitted or empty, exports all providers marked as exportable.
  
  **Example**: `?ids=1,2,3`
</ParamField>

## Response

Returns an array of provider configurations with nested tools and parameters.

<ResponseField name="root" type="array">
  Array of exported provider objects
  
  <Expandable title="Provider Object">
    <ResponseField name="name" type="string" required>
      Human-readable name of the API provider
    </ResponseField>
    
    <ResponseField name="code" type="string" required>
      Unique identifier for the provider (used for upsert operations)
    </ResponseField>
    
    <ResponseField name="baseUrl" type="string" required>
      Base URL for all API requests to this provider
    </ResponseField>
    
    <ResponseField name="authenticationType" type="enum" required>
      Authentication method: `API_KEY`, `BEARER_TOKEN`, `BASIC_AUTH`, `NONE`
    </ResponseField>
    
    <ResponseField name="apiKeyLocation" type="enum">
      Where to send the API key: `HEADER`, `QUERY_PARAM`, `IN_BODY`
    </ResponseField>
    
    <ResponseField name="apiKeyName" type="string">
      Name of the authentication header/parameter (e.g., "Authorization", "x-api-key")
    </ResponseField>
    
    <ResponseField name="apiKeyValue" type="string">
      **Always returns** `<YOUR_API_KEY>` when an API key exists, or `null` if no key is configured. Real API keys are never exposed in exports.
    </ResponseField>
    
    <ResponseField name="isDynamicAuth" type="boolean">
      Whether this provider uses dynamic authentication (token refresh)
    </ResponseField>
    
    <ResponseField name="dynamicAuthUrl" type="string">
      URL endpoint for obtaining fresh authentication tokens
    </ResponseField>
    
    <ResponseField name="dynamicAuthMethod" type="enum">
      HTTP method for dynamic auth: `GET`, `POST`, `PUT`, `PATCH`, `DELETE`
    </ResponseField>
    
    <ResponseField name="dynamicAuthPayloadType" type="enum">
      Format of auth payload: `JSON`, `FORM_DATA`, `XML`
    </ResponseField>
    
    <ResponseField name="dynamicAuthPayloadLocation" type="enum">
      Where to send auth payload: `BODY`, `QUERY_PARAMS`, `HEADERS`
    </ResponseField>
    
    <ResponseField name="dynamicAuthPayload" type="string">
      Template for authentication request payload (JSON/form data)
    </ResponseField>
    
    <ResponseField name="dynamicAuthTokenExtractionPath" type="string">
      JSONPath expression to extract the token from auth response (e.g., "accessJwt", "data.token")
    </ResponseField>
    
    <ResponseField name="dynamicAuthInvalidationKeywords" type="string">
      Comma-separated keywords that trigger token refresh (e.g., "invalid_token,expired_token")
    </ResponseField>
    
    <ResponseField name="customHeaders" type="object">
      Key-value pairs of custom HTTP headers to include in all requests
    </ResponseField>
    
    <ResponseField name="tools" type="array" required>
      Array of tool configurations for this provider
      
      <Expandable title="Tool Object">
        <ResponseField name="name" type="string" required>
          Human-readable tool name
        </ResponseField>
        
        <ResponseField name="code" type="string" required>
          Unique identifier for the tool
        </ResponseField>
        
        <ResponseField name="description" type="string" required>
          Description of what the tool does (shown to AI agents)
        </ResponseField>
        
        <ResponseField name="endpointPath" type="string" required>
          API endpoint path (appended to provider baseUrl). Supports path parameters like `/repos/{owner}/{repo}`
        </ResponseField>
        
        <ResponseField name="httpMethod" type="enum" required>
          HTTP method: `GET`, `POST`, `PUT`, `PATCH`, `DELETE`
        </ResponseField>
        
        <ResponseField name="parameters" type="array" required>
          Array of parameter definitions for this tool
          
          <Expandable title="Parameter Object">
            <ResponseField name="name" type="string" required>
              Parameter name
            </ResponseField>
            
            <ResponseField name="type" type="string" required>
              Parameter type: `STRING`, `NUMBER`, `BOOLEAN`, `OBJECT`, `ARRAY`
            </ResponseField>
            
            <ResponseField name="description" type="string">
              Human-readable description of the parameter's purpose
            </ResponseField>
            
            <ResponseField name="required" type="boolean">
              Whether this parameter is mandatory
            </ResponseField>
            
            <ResponseField name="defaultValue" type="string">
              Default value if not provided by the user
            </ResponseField>
          </Expandable>
        </ResponseField>
      </Expandable>
    </ResponseField>
  </Expandable>
</ResponseField>

## API Key Masking

For security, the `apiKeyValue` field is **always masked** in export responses:

- If an API key is configured: returns `<YOUR_API_KEY>`
- If no API key is configured: returns `null`
- Real API keys are **never** exposed through this endpoint

<Warning>
Do not attempt to import the masked `<YOUR_API_KEY>` value. The import endpoint will skip this placeholder and preserve existing API keys.
</Warning>

## Response Headers

The endpoint includes a content disposition header suggesting a filename:

```
Content-Disposition: attachment; filename=handsai_tools_export.json
```

## Example Request

### Export All Providers

```bash
curl http://localhost:8080/api/export/providers
```

### Export Specific Providers

```bash
curl "http://localhost:8080/api/export/providers?ids=1,3,5"
```

## Notes

- Only providers marked as `exportable` in the database will be included
- The export includes the complete configuration hierarchy
- Tools are exported in the order they are stored in the database
- Parameters maintain their original ordering
- The response is suitable for direct import via the `/api/import/providers` endpoint
